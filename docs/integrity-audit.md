## Phase 2 — Hack / Shortcut Detection (critical findings)

### 1) Automatic bootstrap + demo seeding (DEMO shortcut in runtime path)

**Files**
- `lib/server/bootstrap.js:1`
- `lib/server/admin-data.js:1` (`init()` calls `ensureBootstrapped()`)
- `lib/auth.js:1` (Credentials `authorize()` calls `ensureBootstrapped()`)

**What happens**
- On normal runtime requests/auth, the system will ensure an admin account exists and may seed demo entities (customers, suppliers, materials, orders, worklogs) if empty.

**Why this is a shortcut**
- It makes the system behave like a demo environment. In a “real system”, seed/migration is an explicit operation, not an implicit side effect of login/API calls.
- It can also surprise operators: “data appears” without an intentional seeding action.

**Risk**
- Integrity + operational correctness risk (unexpected writes in production).

**Recommendation**
- Gate seeding behind an explicit command (e.g., `npm run seed:db`) or `NODE_ENV !== "production"`.
- Keep only “ensure default admin exists” if required, but make it explicit and non-demo.

---

### 2) Business ID generation is not concurrency-safe (race + performance shortcut)

**Files**
- `lib/server/admin-data.js:108` (`nextBusinessId()` scans all IDs)
- `app/api/orders/route.js:133` (orderId generated by scanning all existing `ORD-*`)

**What happens**
- IDs like `CUST-###`, `ORD-###`, etc. are computed by reading many documents and taking `max + 1`.

**Why this is a shortcut**
- Not atomic: concurrent creates can generate duplicate IDs.
- Scales poorly: O(n) reads as data grows.

**Risk**
- Duplicate-key errors or (worse) inconsistent behavior under load.

**Recommendation**
- Replace with an atomic counter collection (Mongo `findOneAndUpdate({ _id: 'order' }, { $inc: { seq: 1 }})`).

---

### 3) “Low-stock alert” is encoded as a fake StockLog record (data-model hack)

**Files**
- `app/api/orders/route.js:1` (creates “alert log” with `quantityChanged: 0`)
- `app/api/notifications/route.js:1` (treats `StockLog.quantityChanged === 0` as “alert history”)
- `lib/models/StockLog.js:1`

**What happens**
- The system represents threshold-crossing alerts by inserting `StockLog` entries where `quantityChanged = 0` and `type = "OUT"`.

**Why this is a hack**
- A stock movement log is being reused to represent a notification event (different domain concepts).
- It breaks semantic integrity: an `OUT` stock log with zero quantity is not a real stock movement.

**Risk**
- Confusing analytics/auditing, makes future inventory reconciliation unreliable.

**Recommendation**
- Add a dedicated `Notifications` (or `Alerts`) collection/entity and keep `StockLogs` only for real movements.

---

### 4) Type-check shortcut exists in Next config (even though build runs `tsc`)

**Files**
- `next.config.ts:8` (`typescript.ignoreBuildErrors: true`)

**Why this is a shortcut**
- Next.js build will skip type validation. The repo currently compensates by running `tsc --noEmit` in `npm run build`, but the config itself is still a “bypass”.

**Risk**
- Someone runs `next build` directly (CI/hosting) and silently ships TS errors.

**Recommendation**
- Set `ignoreBuildErrors: false` and keep `tsc --noEmit` as additional guard if desired.

---

### 5) UI suggests user-supplied IDs, but backend silently ignores them (fake UX / mismatch)

**Files**
- `app/(admin)/customers/new/page.tsx:1` (has “Customer ID” input)
- `app/(admin)/inventory/raw-materials/new/page.tsx:1` (has “Material ID” input)
- `lib/validations/entities.js:1` (customer/material create schemas do not include `id`)

**What happens**
- The UI collects ID fields, but Zod schemas strip unknown keys, so the backend generates IDs anyway.

**Why this is a shortcut**
- The system appears to accept an ID, but actually discards it.

**Risk**
- User confusion, inconsistent records, and “fake” behavior during demos/viva.

**Recommendation**
- Either (A) remove ID inputs from UI, or (B) explicitly support admin-provided IDs with strict validation + uniqueness.

---

### 6) Silent error swallowing in client components (weak error handling)

**Files**
- `features/orders/order-assignment-panel.tsx:1` (`.catch(() => null)`)
- `features/orders/employee-progress-panel.tsx:1` (`.catch(() => null)`)
- `features/estimation/estimation-form.tsx:1` (`.catch(() => null)`)

**Why this is weak**
- Failures become invisible; UI may misleadingly render “no data” rather than “could not load”.

**Recommendation**
- Convert to explicit error UI states, or at least log/report errors.

---

### 7) Risky defaults / schema semantics that can create “silent privilege”

**Files**
- `lib/models/User.js:1`

**What happens**
- `role` defaults to `"admin"` at the schema level.
- `id` is optional (`sparse: true`), which can leave records without a stable business id unless every creation path enforces it.

**Why this is risky**
- Any future code path that inserts a user without explicitly setting `role` could accidentally create an admin.

**Recommendation**
- Make `role` required with a safe default (or no default) and enforce role assignment only in controlled flows.
- Consider making `id` required for all users (or ensure it is always set server-side on create).

## Phase 3 — Deep Flow Verification (end-to-end reality check)

Validated via:
- Script: `scripts/validate-ucs.cjs`
- Command executed successfully: `npm.cmd run validate:ucs` (PASS)

### Flow 1: Create Order → check stock → deduct → create StockLog

**Implementation**
- `app/api/orders/route.js` uses a Mongo session transaction:
  - validates customer/material existence
  - validates sufficient quantity
  - decrements stock atomically via `findOneAndUpdate({ quantity: { $gte: qty }}, { $inc: { quantity: -qty }})`
  - writes `StockLog` OUT entries
  - persists `Order` with `materialsUsed.priceAtTime`

**Integrity verdict**
- Core logic is real (not faked) and is transactional.
- **Still risky** due to non-atomic ID generation and alert-as-stocklog hack.

### Flow 2: WorkLog → submit → approve → calculate wage

**Implementation**
- Submit: `app/api/work-logs/route.js` → `lib/server/admin-data.js` (`createWorkLog()`)
- Employee can only submit if assigned to order (assignment check in `createWorkLog()`).
- Approve: `lib/server/admin-data.js` (`updateWorkLog()`):
  - only admin can approve/reject
  - sets `approvedBy`, `approvedAt`
  - wage = `hourlyRate * hoursWorked` when approved

**Integrity verdict**
- Real validation and wage calculation exist.

### Flow 3: Order → complete → generate invoice

**Implementation**
- Completion: `app/api/orders/[id]/route.js:1` enforces allowed lifecycle transitions.
- Invoice: `lib/server/admin-data.js` (`createInvoice()`) requires `order.status === "completed"`.
- Invoice total = material cost (`quantityUsed * priceAtTime`) + sum(approved worklog wages).

**Integrity verdict**
- Guardrails match the spec and real-world expectation.

### Flow 4: Payment → update remaining balance

**Implementation**
- `lib/server/admin-data.js` (`createPayment()`):
  - validates `amount <= remainingBalance`
  - uses transaction + recalculation

**Integrity verdict**
- Overpayment prevented; balance maintained.

### Flow 5: Low stock → trigger alert

**Implementation**
- Event-driven in `app/api/orders/route.js:1` after decrement.
- Notification UI reads low materials from `app/api/notifications/route.js:1`.

**Integrity verdict**
- Works end-to-end, but the “alert history” storage is a semantic hack (see Phase 2 #3).

## Phase 5 — Edge Cases & Race Conditions

### Duplicate entries
- Customer duplicates are checked in code (`phone/email`) but are **not backed by unique DB indexes** on `phone` or `email`.
  - Risk: concurrent creates can still duplicate.
  - Files: `lib/server/admin-data.js:170`, `lib/models/Customer.js:1`

### Date fields stored as strings (weak data typing)
- `WorkLog.workDate` and `Payment.paymentDate` are stored as strings (`lib/models/WorkLog.js:1`, `lib/models/Payment.js:1`).
- Risk: inconsistent formats, hard queries/ranges, timezone confusion.
- Recommendation: store as `Date` in DB and normalize at API boundaries.

### Authorization integrity
- API auth is enforced twice (middleware + per-route session checks). After middleware adjustment:
  - Unauthorized API calls return JSON 401/403 rather than redirect (correct for API consumers).
  - Files: `middleware.js:1`, `lib/api/request.js:1`

### Concurrency
- Stock decrement is concurrency-safe per material line item.
- **Business ID generation is not concurrency-safe** (highest integrity risk remaining).
